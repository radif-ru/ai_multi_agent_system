# Спринт 05. OCR: рефакторинг и расширение на одиночные изображения

- **Источник:** запрос пользователя (расширение OCR за пределы PDF) + наблюдение по итогам спринта 03 (OCR прошит внутри `read_document._read_pdf`).
- **Ветка:** `feature/05-ocr` (от `main`; см. `process.md` §2, п.2).

## 1. Цель спринта

Сейчас OCR живёт внутри `app/tools/read_document.py::_read_pdf` (строки 105–230 на момент открытия спринта): сборка путей картинок, выбор языка, кеш `<file>.ocr.txt` и сам вызов `pytesseract` — всё в императивном блоке посреди извлечения PDF. Для произвольных фото в `tmp/` OCR не применяется — единственный путь работы с изображением (`DescribeImageTool`) идёт через Vision-LLM, что для скринов/чеков/документов с плотным текстом медленнее и даёт пересказ вместо точной транскрипции.

Спринт выносит OCR-логику в самостоятельный сервис, добавляет отдельный tool `ocr_image` для одиночных изображений и фиксирует правила выбора OCR vs Vision в skill. Никаких изменений в агентном цикле, событийной модели или модуле Users — спринт сознательно узкий и независимый.

## 2. Скоуп и non-goals

### В скоупе

- Новый сервис `app/services/ocr.py` с чистым контрактом распознавания текста с изображений.
- Рефакторинг `app/tools/read_document.py::_read_pdf` — делегирование OCR в сервис без изменения формата вывода и поведения кеша.
- Новый tool `app/tools/ocr_image.py` для произвольных изображений в `tmp/` (по аналогии с `DescribeImageTool`).
- Новый skill `_skills/ocr/SKILL.md` с правилами выбора `ocr_image` / `describe_image` / связки.
- Обновление конфигурации (`app/config.py`): вынесение магических значений (`ocr_default_lang`, `ocr_min_text_threshold`).
- Обновление документации: `_docs/tools.md`, `_docs/skills.md`, `_docs/architecture.md` §3, `_docs/current-state.md` §1, `_docs/roadmap.md`.

### Вне скоупа (non-goals)

- Замена `pytesseract` на другой OCR-движок (PaddleOCR, EasyOCR, облачные API).
- Авто-OCR на upload-pipeline фотографий (прозрачное подмешивание текста к описанию вне ведома агента) — это размывает контракт инструментов, оставляем выбор за skill'ом.
- Переход на хеш-кеш OCR вместо файлового — текущий `<stem>.ocr.txt` достаточен, миграция не требуется.
- Многостраничные не-PDF документы (TIFF и т. п.) — не поддерживаем сейчас.
- Любые изменения в `DescribeImageTool` (vision) и в Vision-сервисе.
- Изменения событийной шины и модуля Users — это спринт 04.

## 4. Этап 1. Сервис OCR

Выделить чистый сервис без сайд-эффектов на pdf-структуру, чтобы дальше переиспользовать его из tool'а одиночных изображений.

### Задача 1.1. Выделить OCR-логику в `app/services/ocr.py`

- **Статус:** ToDo
- **Приоритет:** high
- **Объём:** S
- **Зависит от:** —
- **Связанные документы:** `_docs/architecture.md` §3 (компоненты), `_docs/tools.md` (раздел про `read_document`).
- **Затрагиваемые файлы:** новый `app/services/ocr.py`, `app/tools/read_document.py`, `app/config.py`, новый `tests/services/test_ocr.py`, обновления `tests/tools/test_read_document.py` (если есть).

#### Описание

1. `OcrService` (или модульные функции — на усмотрение исполнителя, с явным контрактом):
   - `extract_text(image_paths: Sequence[Path], lang: str | None = None) -> str` — возвращает склеенный текст по картинкам; `lang=None` → автоопределение (`rus+eng` если `rus` доступен, иначе `eng`).
   - `get_default_lang() -> str` — кешированное определение языков через `pytesseract.get_languages`.
   - Кеш OCR: `<image_stem>.ocr.txt` рядом с файлом для одиночных картинок; для совместимости с PDF — путь кеша задаётся параметром (см. п. 2).
2. `read_document._read_pdf` после рефакторинга **только** собирает пути картинок и делегирует распознавание сервису. Формат итогового вывода (`[OCR текст из изображений:]\n...`) и кеш-файл `<pdf_stem>.ocr.txt` сохраняются один-в-один — это проверяется regression-тестом.
3. Если `pytesseract` или `PIL` не установлены — сервис возвращает пустую строку и пишет `WARNING` в лог (мягкая деградация, как сейчас).
4. Конфиг: добавить в `app/config.py`:
   - `ocr_default_lang: str = "rus+eng"` — язык по умолчанию (используется и сервисом, и tool'ом из задачи 2.1).
   - `ocr_min_text_threshold: int = 100` — порог `len(text) < N` для запуска OCR в PDF (раньше захардкожен в `_read_pdf`).
   Существующие флаги `read_document_*` не трогать.

#### Definition of Done

- [ ] `app/services/ocr.py` создан; `tests/services/test_ocr.py` покрывает: успешное извлечение (мок `pytesseract`), отсутствие `pytesseract` (мягкая деградация), кеш-хит, кеш-промах, выбор языка по `get_languages` (rus есть / нет).
- [ ] `read_document._read_pdf` использует сервис; добавлен regression-тест на стабильность вывода и кеша `<pdf_stem>.ocr.txt`.
- [ ] `pytest -q` зелёный.
- [ ] **Документация обновлена**: `_docs/architecture.md` §3 — упомянуть новый сервис; `_docs/tools.md` — пометка «OCR в `read_document` делегируется в `app/services/ocr.py`». До коммита кода.
- [ ] **Тесты добавлены / обновлены**: см. выше.
- [ ] `git status` чист.

## 5. Этап 2. Tool `ocr_image` и skill

Дать агенту явный инструмент для распознавания текста с одиночных изображений и зафиксировать правила выбора между OCR и Vision.

### Задача 2.1. Новый tool `ocr_image` для произвольных изображений в `tmp/`

- **Статус:** ToDo
- **Приоритет:** high
- **Объём:** S
- **Зависит от:** Задача 1.1
- **Связанные документы:** `_docs/tools.md`, `_docs/agent-loop.md` (контракт tools).
- **Затрагиваемые файлы:** новый `app/tools/ocr_image.py`, `app/main.py`, `app/console_main.py` (регистрация в `ToolRegistry`), новый `tests/tools/test_ocr_image.py`.

#### Описание

Tool по аналогии с `DescribeImageTool`:

1. `name = "ocr_image"`, аргументы: `image_path: str` (обязательный), `lang: str | None` (опциональный, по умолчанию — `settings.ocr_default_lang`).
2. Валидация пути — как в `DescribeImageTool`: только внутри `tmp_dir`, расширения `.jpg/.jpeg/.png/.gif/.webp`, защита от `..`.
3. Делегирует распознавание сервису из 1.1; возвращает извлечённый текст, обрезанный `truncate_output` до разумного лимита (8000 символов — сравнимо с `read_document`).
4. Если результат пустой — возвращает явное сообщение `"OCR не нашёл текста на изображении"`, а не пустую строку (агенту проще принять решение).
5. Кеш `<image_stem>.ocr.txt` — управляется сервисом из 1.1.

#### Definition of Done

- [ ] Tool зарегистрирован в `ToolRegistry` обоих entry-points (`app/main.py`, `app/console_main.py`).
- [ ] Unit-тесты покрывают: успешное распознавание (мок сервиса), пустой результат, валидация пути (вне `tmp/`, `..`, не-изображение, несуществующий файл), повторный вызов читает из кеша.
- [ ] `pytest -q` зелёный.
- [ ] **Документация обновлена**: `_docs/tools.md` — раздел про `ocr_image` (контракт, когда применять); `_docs/current-state.md` §1 — отметка о новом tool.
- [ ] **Тесты добавлены / обновлены**: см. выше.

### Задача 2.2. Skill `_skills/ocr/SKILL.md` с правилами выбора OCR vs Vision

#### Описание

Зафиксировать для агента правила выбора между `ocr_image`, `describe_image` и связкой обоих:

1. **Только `ocr_image`** — скрин переписки/документа/чека/таблицы, где задача — точно прочитать текст; обычное описание не нужно.
2. **Только `describe_image`** — фото сцены/объекта без текста или с малозначимой подписью.
3. **`describe_image` + `ocr_image`** — фото со смешанным содержимым (например, фото объявления на стене), когда нужно и описать сцену, и точно процитировать текст.
4. Явно отметить: для PDF агент по-прежнему вызывает `read_document` — внутри он сам решит, нужен ли OCR.

Skill короткий (по образцу существующих в `_skills/`), без кода — только правила и примеры формулировок запроса.
