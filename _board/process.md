# Процесс выполнения спринтов и задач

Единый свод правил: легенды, правила работы со спринтами и задачами, шаблоны, пошаговый алгоритм от открытия спринта до его закрытия. Документ **статичен** — меняется только при смене правил процесса. Любой ИИ или человек, открывший этот файл, получает полный набор инструкций для единообразного заполнения документации проекта.

Связанные документы:

- `_board/plan.md` — индекс спринтов (только таблицы, без правил).
- `_board/progress.txt` — ad-hoc заметки и чек-листы приёмки.
- `_board/sprints/<NN>-<short-name>.md` — файлы спринтов.
- `_docs/instructions.md` — правила разработки (git-дисциплина, стиль, тесты, документация).
- `CLAUDE.md` — поведенческие гайдлайны LLM-агента.

---

## 1. Легенды

### 1.1 Статусы задач

| Статус | Описание |
|--------|----------|
| `ToDo` | Задача готова к работе (Definition of Ready выполнен). |
| `Progress` | Задача в работе. Одновременно в `Progress` — **ровно одна** задача на всей доске. |
| `Done` | Задача завершена, Definition of Done выполнен. |
| `Blocked` | Задача заблокирована (нужны внешние действия / разрешение зависимостей). |

### 1.2 Статусы спринтов

| Статус | Описание |
|--------|----------|
| `Active` | Спринт идёт, по нему есть задачи в `Progress` и/или `ToDo`. |
| `Closed` | Все задачи спринта в `Done`, Acceptance Criteria закрыты. |
| `Cancelled` | Спринт остановлен до закрытия (фиксируется причина). |

### 1.3 Приоритеты задач

| Приоритет | Когда использовать |
|-----------|--------------------|
| `high` | Основной функционал, критичный баг, блокер других задач. |
| `medium` | Улучшения, оптимизации, техдолг. |
| `low` | Nice-to-have, косметика, документация без блокеров. |

### 1.4 Объём задач

| Объём | Примерная длительность |
|:-----:|:-----------------------|
| XS | < 1 часа |
| S | 1–4 часа |
| M | 4–8 часов |
| L | 1–2 дня |
| XL | > 2 дней (разбить на задачи поменьше) |

## 2. Правила работы со спринтами

1. **Один активный спринт за раз.** Новый спринт открывается только после закрытия предыдущего. Исключение — заранее согласованный параллельный «фоновый» спринт (помечается явно в индексе `_board/plan.md`).
2. **Новый спринт = новая ветка.** Каждый новый спринт открывается в отдельной ветке `feature/<NN>-<short-name>` от актуальной `main`. Имя ветки фиксируется в шапке файла спринта и в индексе спринтов. Коммиты спринта идут только в эту ветку; merge в `main` делает пользователь сам — после закрытия спринта. Исключение — спринт 00 (инфраструктурный, `main`).
3. **Каждому спринту — отдельный файл с фиксированной структурой.** При старте создаётся `_board/sprints/<NN>-<short-name>.md` (двузначный `<NN>` с leading zero) по шаблону §4.1. **Декомпозиция:** спринт → этапы → задачи. Этапы нумеруются с 1, идут по порядку выполнения. Внутри этапа задачи нумеруются с 1 и имеют локальный ID `<stage>.<task>`.
4. **Зависимости задач.** Поле «Зависит от» у задачи может ссылаться на задачи внутри этого же этапа, других этапов того же спринта или задач закрытых ранее спринтов. Нельзя зависеть от задачи **другого активного** спринта (см. п.1).
5. **История спринтов неизменна.** Закрытые спринты остаются в `_board/sprints/` без правок — только секция «История изменений спринта» дополняется датой закрытия при переводе статуса. Если в закрытом спринте есть ссылки на старую структуру или нумерацию документации — они остаются как есть (архив).
6. **Индекс обновляется синхронно.** При открытии/закрытии спринта актуализируется `_board/plan.md` § «Индекс спринтов».
7. **Источник новых спринтов** — ТЗ пользователя + кандидаты из `_docs/roadmap.md` и `_docs/current-state.md`.

## 3. Правила работы с задачами

1. Задачи берутся **сверху вниз** в файле текущего спринта — сначала задачи этапа 1, затем этапа 2, и т. д. Нельзя брать новую задачу, пока её зависимости (поле «Зависит от») не в `Done`.
2. В `Progress` одновременно — **ровно одна задача на всей доске**.
3. **Нумерация задач и формат коммитов:**
   - **Локальный ID** в файле спринта — двухчастный `<stage>.<task>`. `<stage>` — номер этапа в спринте (1, 2, 3, …); `<task>` — номер задачи в этапе (1, 2, 3, …). Нумерация начинается с 1 и **не совпадает** с номером пункта в оглавлении. Примеры: `1.1`, `1.2`, `2.1`, `3.6`.
   - Sprint-id в локальном ID **не дублируется**.
   - **Коммит смены статуса** — формат `chore(plan): начать задачу <NN>.<stage>.<task>` или `chore(plan): закрыть задачу <NN>.<stage>.<task>`, где `<NN>` — двузначный sprint-id (совпадает с префиксом имени файла спринта). Примеры: `chore(plan): начать задачу 01.2.4`, `chore(plan): закрыть задачу 03.4.7`.
4. **Новые задачи можно добавлять** в активный спринт по ходу работы (это не нарушает «историю неизменности» — запрет касается только **закрытых** спринтов). Новые задачи получают следующий свободный ID в своём этапе; номера ранее закрытых задач не переиспользуются.
5. **Порядок выполнения и исключения.** Задачи выполняются по порядку номеров внутри этапа (1.1 → 1.2 → 1.3). Исключения:
   - **Критичный баг** — можно выполнять вне очереди; причина кратко указывается в поле «Зависит от» (например, «баг блокирует тестирование»).
   - **Зависимость от другой задачи** — если задача A зависит от B (указано в «Зависит от»), а B ещё не `Done`, можно либо подождать B, либо явно запросить у пользователя разрешение на изменение порядка.
6. **Все коммиты спринта идут в его рабочую ветку** (`feature/<NN>-<short-name>`), указанную в шапке файла спринта. Язык сообщений — по общему правилу проекта (`_docs/instructions.md` §0).

## 4. Шаблоны

### 4.1 Шаблон файла спринта

Структура файла `_board/sprints/<NN>-<short-name>.md` фиксирована. Обязательные секции в указанном порядке:

```markdown
# Спринт <NN>. <Название>

- **Источник:** ТЗ пользователя / `_docs/roadmap.md` § ... / `_docs/current-state.md` § ...
- **Ветка:** `feature/<NN>-<short-name>` (от `main`; см. §2, п.2).
- **Открыт:** YYYY-MM-DD
- **Закрыт:** — (или YYYY-MM-DD при закрытии)

## 1. Цель спринта

1–2 абзаца: чего достигаем и почему это важно сейчас.

## 2. Скоуп и non-goals

### В скоупе
- Явный список областей, которые трогаем.

### Вне скоупа (non-goals)
- Явный список областей, которых НЕ касаемся, чтобы ограничить scope creep.

## 3. Acceptance Criteria спринта

Крупные проверяемые результаты, которые должны быть выполнены при закрытии спринта. AC спринта отличается от DoD отдельных задач: AC перекрывает эффект всего спринта целиком.

- [ ] Результат 1.
- [ ] Результат 2.
- [ ] Все задачи спринта — `Done`, сводная таблица актуальна.

## 4. Этап 1. <Название этапа>

1–3 строки о цели этапа.

### Задача 1.1. <Название задачи>

(См. §4.2 «Шаблон задачи».)

### Задача 1.2. <Название задачи>

…

## 5. Этап 2. <Название этапа>

### Задача 2.1. …

## N. Этап K. <Название>

### Задача K.1. …

## (N+1). Риски и смягчение

| # | Риск | Смягчение |
|---|------|-----------|
| 1 | …    | …         |

## (N+2). Сводная таблица задач спринта

| #   | Задача | Приоритет | Объём | Статус | Зависит от |
|-----|--------|:---------:|:-----:|:------:|:----------:|
| 1.1 | …      | …         | …     | ToDo   | —          |

> Обновляется при каждом переходе статуса и при добавлении/удалении задач.

## (N+3). История изменений спринта

- **YYYY-MM-DD** — спринт открыт, ветка `feature/<NN>-<short-name>` создана от `main`.
- **YYYY-MM-DD** — закрыта задача X.Y: <кратко>.
- **YYYY-MM-DD** — спринт закрыт (или Cancelled с причиной).
```

**Правила файла спринта:**

- Секции 1–3 (Цель, Скоуп, AC) — пишутся при открытии спринта, в идеале не меняются.
- Секции «Этапы» (4..N) — обязательно минимум один этап, у каждого — отдельная `##`-секция и минимум одна задача внутри.
- Секции (N+1)..(N+3) — обязательные вне зависимости от размера спринта.
- **Задачи внутри этапа** — оформляются по §4.2 «Шаблон задачи».
- **Порядок этапов** отражает порядок выполнения: нельзя начинать задачи этапа K, пока не закрыты все блокирующие зависимости из этапов < K.
- **Закрытие спринта:** все задачи — `Done`, все пункты AC проставлены, запись в «Истории изменений спринта» добавлена, `_board/plan.md` актуализирован (см. §10).

### 4.2 Шаблон задачи

Используется внутри файла спринта (`### Задача <stage>.<task>. <название>`):

```markdown
### Задача <stage>.<task>. <Краткое название>

- **Статус:** ToDo
- **Приоритет:** high | medium | low
- **Объём:** XS | S | M | L | XL
- **Зависит от:** Задача X.Y, Задача X.Z или —
- **Связанные документы:** `_docs/<file>.md` § <раздел>; …
- **Затрагиваемые файлы:** `app/<path>.py`, … (предполагаемые).

#### Описание

Что и зачем нужно сделать. Контекст, ссылки на источник (`_docs/current-state.md` § X.Y, либо запрос пользователя).

Конкретные шаги (если применимо):

1. Шаг 1.
2. Шаг 2.
3. …

#### Definition of Done

- [ ] Условие 1 (проверяемо: smoke / тестом / инспекцией).
- [ ] Условие 2.
- [ ] **Документация обновлена** (обязательно для задач, меняющих поведение системы): соответствующий документ в `_docs/` обновлён до коммита кода. Для чисто-документационных задач — `n/a`.
- [ ] **Тесты добавлены / обновлены** (обязательно для задач, меняющих код в `app/`): хотя бы один unit-тест на новое поведение, `pytest -q` зелёный. Для чисто-документационных задач (только `_docs/`, `_board/`, `README.md`, `_skills/`, `_prompts/`) — `n/a`.
- [ ] `git status` чист, артефакты не закоммичены.
```

## 5. Подготовка к сессии

Выполняется один раз на сессию работы:

1. `git status` пуст, ветка актуальна (`git pull`).
2. Активировано виртуальное окружение: `source .venv/bin/activate`.
3. Зависимости актуальны: `pip install -r requirements.txt` (если был добавлен/обновлён модуль).
4. Ollama запущена: `ollama list` отвечает; нужные модели на месте (`qwen3.5:4b`, `nomic-embed-text`, `llava:7b` или `gemma3:4b`).
5. **На активной ветке спринта** (см. `_board/plan.md` § «Индекс спринтов» → колонка «Ветка»).

---

## 6. Открытие спринта

**Выполняется по запросу пользователя** (кроме случаев, когда пользователь явно попросил открывать/закрывать несколько спринтов автоматически).

1. **Создать ветку** от `main`:
   ```bash
   git checkout main && git pull
   git checkout -b feature/<NN>-<short-name>
   ```

2. **Создать файл** `_board/sprints/<NN>-<short-name>.md` по шаблону §4.1. Заполнить секции «Цель», «Скоуп и non-goals», «Acceptance Criteria», минимум один этап с минимум одной задачей.

3. **Обновить** `_board/plan.md` — добавить спринт в таблицу «Активные» (статус `Active`, дата открытия).

4. **Закоммитить**:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md
   git commit -m "chore(plan): открыть спринт <NN> — <название>"
   ```
   > Формулировка **строго «открыть»**, не «начать» и не «создать».

> **Push и merge в `main` делает пользователь сам** — по явному запросу, не автоматически.

## 7. Выполнение задачи

Универсальный алгоритм закрытия **одной** задачи из активного спринта. Содержание задачи и её DoD берутся из `_board/sprints/<active>.md` и связанных документов в `_docs/`.

### 7.1 Прочитать документацию

Обязательный шаг перед началом любой задачи:

1. `CLAUDE.md` — поведенческие гайдлайны LLM-агента.
2. `_docs/README.md` — навигация по проектной документации.
3. `_docs/current-state.md` — есть ли запись про затрагиваемую область (потенциальные подводные камни, легаси).
4. `_docs/instructions.md` — правила разработки (git, стиль, async, ошибки, секреты, тесты, документация). Перечитывать до автоматизма.
5. `_board/plan.md` — индекс спринтов: какой спринт активен и какая у него ветка.
6. `_board/sprints/<active>.md` — выбираемая задача и её соседи (зависимости).
7. Документы из поля «Связанные документы» выбранной задачи (например, `_docs/agent-loop.md` для задач по Executor; `_docs/memory.md` — для задач по `sqlite-vec`; `_docs/tools.md` — для нового tool).

При расхождении документа и кода — приоритет у кода, документ обновляется после задачи отдельным `docs(...)`-коммитом.

### 7.2 Выбрать задачу

1. Открыть `_board/plan.md` → найти активный спринт → открыть его файл `_board/sprints/<NN>-<short-name>.md`.
2. Найти **первую сверху** задачу со статусом `ToDo` (сначала задачи этапа 1, затем этапа 2, …).
3. Проверить поле «Зависит от»: все указанные задачи должны быть `Done`. Если нет — взять задачу без невыполненных зависимостей.
4. Если требуются внешние действия (ТЗ, доступы, решения от пользователя), которых нет, — пометить задачу `Blocked` с причиной, остановиться.

### 7.3 Начать задачу (`Progress`)

1. В файле спринта у выбранной задачи изменить `**Статус:** ToDo` → `**Статус:** Progress`.
2. Обновить «Сводную таблицу задач спринта» в этом же файле (колонка «Статус»).
3. Закоммитить:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): начать задачу <NN>.<stage>.<task>"
   # пример: chore(plan): начать задачу 03.4.7
   ```

> В статусе `Progress` — **ровно одна задача на всей доске**.

### 7.4 Подготовить контекст

1. Прочитать «Описание» и «Definition of Done» задачи целиком.
2. Открыть в редакторе файлы из секции «Затрагиваемые файлы» (или предполагаемые по описанию).
3. Если задача связана с конкретной подсистемой — открыть параллельно соответствующий документ из `_docs/`:
   - конфиг / логирование → `stack.md` §9 + `architecture.md` §3.2–3.3;
   - LLM-клиент → `architecture.md` §3.4;
   - агентный цикл → `agent-loop.md`;
   - tools → `tools.md`;
   - память → `memory.md`;
   - skills / prompts → `skills.md` / `prompts.md`;
   - команды бота → `commands.md`.
4. Сформулировать (для себя или для LLM-помощника) короткий план шагов.

### 7.5 Реализовать

Следуем правилам `_docs/instructions.md` (минимализм, хирургические правки, цель → проверка):

1. **Самый минимальный диф**, решающий проблему.
2. **Не править соседний код** без явной задачи на это.
3. **Не добавлять «гибкости на будущее»** — реализуй ровно то, что нужно.
4. **Соблюдай существующий стиль** (импорты, нейминг, структура файлов).
5. Если встретился несвязанный баг или мёртвый код — записать в `_docs/current-state.md` §2, **не править** в этой задаче.

Цикл «правка → проверка»:

1. Внести правку.
2. Прогнать smoke-проверки (см. §7.6).
3. Если что-то поломалось — откатить и переделать.
4. **Для задач, меняющих код в `app/` — обязательно добавить или обновить тесты на новое поведение до зелёного состояния** (см. §7.6). Чисто-документационные задачи от этого правила освобождены — в DoD ставится `n/a`.

### 7.6 Проверка изменений

Минимальный smoke-test (всегда):

```bash
# 1) Импорт пакета не сломан
python -c "import app; print('OK')"

# 2) Сборка приложения (точка входа) — если затронуты handlers / main / services
python -c "import asyncio; from app.main import main; print(main)"

# 3) Один сценарий вручную (для задач, меняющих UX) — в реальном Telegram:
python -m app
# отправить /start и одну задачу, затем Ctrl+C
```

Целевые проверки (когда применимо):

- **Конфигурация** — `python -c "from app.config import Settings; print(Settings().ollama_default_model)"`.
- **LLM-клиент** изменён — `pytest tests/services/test_llm_client.py -q`.
- **Агентный цикл** изменён — `pytest tests/agents -q`.
- **Tool** изменён — `pytest tests/tools/test_<name>.py -q`.
- **`SemanticMemory` / `Archiver`** изменены — `pytest tests/services/test_memory.py tests/services/test_archiver.py -q` + ручная проверка `/new` в Telegram.
- **Handlers** изменены — `pytest tests/adapters/telegram -q` + ручная проверка соответствующей команды.
- **Middleware / Errors** — намеренно падающий хендлер локально → бот должен ответить «Что-то пошло не так», polling не падает.
- **`.env` / секреты** — `git status` не показывает `.env`; `git log -p | grep TELEGRAM_BOT_TOKEN=` возвращает только примеры из `.env.example`.

Полный прогон тестов:

```bash
pytest -q                                      # все тесты
pytest tests/<затронутый-модуль> -q -v         # детально по подсистеме
pytest --cov=app --cov-report=term-missing     # покрытие (если установлен pytest-cov)
```

### 7.7 Проверить Definition of Done

1. Открыть в файле спринта раздел «Definition of Done» выбранной задачи.
2. По каждому пункту — отметить чекбоксы `[x]` (выполнено).
3. Если хотя бы один пункт не закрыт — вернуться к §7.5.
4. **Обязательное обновление документации.** Если задача меняет поведение системы — обновить соответствующий документ в `_docs/` (см. §8). Это делается **до** коммита кода.
5. Прогнать `git status` — никаких лишних файлов (`*.log`, `.env`, `__pycache__/`, `data/*.db`).

### 7.8 Зафиксировать результат в git

**Перед каждым коммитом задачи (кроме `chore(plan): начать/закрыть задачу ...`) прогнать `pytest -q` и убедиться, что он зелёный.** Если тесты падают (даже не связанные с задачей) — сначала чинятся, и только потом коммит. См. `_docs/instructions.md` §7.2.

Коммиты — атомарные, [Conventional Commits](https://www.conventionalcommits.org/), в императиве. Язык сообщения — по общему правилу проекта (`_docs/instructions.md` §0):

```bash
pytest -q                              # обязательно: должно быть зелёным
git add <изменённые файлы>
git commit -m "feat(<scope>): <что сделано>"
# или: fix(...) / refactor(...) / docs(...) / test(...) / chore(...)
```

Типичные scope этого проекта:

- `feat(agents): добавить парсер JSON ответа модели`
- `feat(tools): реализовать calculator с безопасным AST-парсингом`
- `feat(memory): добавить SemanticMemory на sqlite-vec`
- `feat(handlers): добавить команду /new с архивированием в sqlite-vec`
- `fix(llm): обработать httpx.TimeoutException → LLMTimeout`
- `refactor(executor): вынести построение initial messages в отдельный метод`
- `docs(architecture): уточнить поток /new в §5`
- `test(tools): покрыть calculator случаем деления на ноль`
- `chore(plan): закрыть задачу 03.4.7`

### 7.9 Перевести задачу в `Done`

1. В файле спринта изменить `**Статус:** Progress` → `**Статус:** Done`.
2. Обновить «Сводную таблицу задач спринта».
3. Отметить все чекбоксы DoD как `[x]`.
4. Дописать строку в «История изменений спринта»: дата + краткое описание + ссылка на коммит/коммиты.
5. **Важно:** статус задачи обновляется **везде**, где она упоминается (заголовок, сводная таблица, DoD).
6. Коммит:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): закрыть задачу <NN>.<stage>.<task>"
   ```

## 8. Обновление документации `_docs/`

**Обязательный шаг для задач, меняющих поведение системы.** Выполняется **до** коммита кода (см. §7.7 и §7.8).

1. Найти соответствующий документ (`commands.md`, `architecture.md`, `agent-loop.md`, `memory.md`, `tools.md`, `skills.md`, `prompts.md`, `stack.md`, `testing.md`, `vision-models.md`).
2. Внести правку.
3. Если изменилась структура каталога или появился новый документ — обновить `_docs/README.md` § «Навигация» и `_docs/project-structure.md`.
4. Если найден новый легаси-нюанс — добавить в `_docs/current-state.md` §2.
5. Коммит `docs(<scope>): ...` — **отдельным коммитом** перед кодовым коммитом.

Чисто-документационные задачи (правят только `_docs/`, `_board/`, `README.md`, `.env.example`, `_skills/`, `_prompts/`) от требования кодовых тестов освобождены — в DoD ставится `n/a` напротив пункта про тесты.

## 9. Обновление `_board/progress.txt`

`progress.txt` — ad-hoc заметки. Формальные правила процесса и планирования туда **не пишутся** (они здесь, в `process.md`). Туда идёт то, что не ложится в формальные документы, но полезно держать перед глазами.

Когда обновлять:

1. **После закрытия спринта** — добавить запись в «Заметки сессии» с датой и коротким итогом (пример: «2026-05-01 — Спринт 03 закрыт. Все задачи выполнены»).
2. **По ходу работы**, если всплыл нюанс окружения или состояния чек-листа приёмки, который ещё нельзя зафиксировать в `_docs/current-state.md` (например, «покрытие не измерено, pytest -q зелёный на 256 тестах»).
3. **При изменении статуса пунктов чек-листа приёмки спринта** — проставить `[+]` / `[~]` / `[ ]` напротив пункта.

Крупные найденные нюансы (баги, легаси, архитектурные ограничения) идут в `_docs/current-state.md` §2, а не в `progress.txt`.

## 10. Закрытие спринта

**Решение о закрытии принимает пользователь** (кроме случаев явного запроса «закрывай автоматически, продолжай несколько спринтов подряд»).

Когда все задачи спринта — `Done`:

1. **По умолчанию** — спросить пользователя: «Все задачи спринта выполнены. Закрыть спринт <NN>?»
2. **Обновить файл спринта** `_board/sprints/<NN>-<short-name>.md`:
   - Поле **Закрыт:** YYYY-MM-DD.
   - Статус в `_board/plan.md` — `Closed`.
   - В «Истории изменений спринта» — строка о закрытии.
3. **Обновить** `_board/plan.md`:
   - Перенести спринт из «Активные» в «Закрытые».
   - Заполнить колонки: «Открыт», «Закрыт», «Статус».
4. **Обновить** `_board/progress.txt` — добавить короткую запись в «Заметки сессии» (см. §9).
5. **Закоммитить**:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md _board/progress.txt
   git commit -m "chore(plan): закрыть спринт <NN> — <название>"
   ```

> **Merge в `main` делает пользователь сам** — только по явному запросу. В коммит закрытия **не добавлять задачи следующих спринтов**.

7. Коммит правок кода (`feat/fix/refactor/test/chore`).
