# `_skills/` — markdown-скиллы для агента

Каталог содержит **скиллы** — markdown-инструкции, которые агент динамически подгружает в свой контекст по необходимости. Полная спецификация — `_docs/skills.md`.

## Зачем

- В отличие от `tool` (детерминированный код в `app/tools/`), skill — это **знание о том, как решать класс задач**: последовательность шагов, формат ответа, эвристики.
- Описание скилла (одна строка) попадает в системный промпт агента; полное тело подгружается только когда агент через tool `load_skill` решит его использовать.
- Это позволяет масштабировать набор «навыков» без перекомпиляции кода и без раздувания системного промпта.

## Структура

```
_skills/
├── README.md                     # этот файл (правила, шаблон)
├── example-summary/
│   └── SKILL.md                  # legacy формат: Description: ...
├── bash-linux/
│   └── SKILL.md                  # YAML frontmatter формат
├── bash-pro/
│   └── SKILL.md
├── weather/
│   └── SKILL.md
├── ocr/
│   └── SKILL.md
└── ...
```

Имя подпапки — **это имя скилла** (kebab-case), по которому агент его вызывает: `load_skill("example-summary")`.

Поддерживаются два формата `SKILL.md`:
1. **Legacy:** `Description: <текст>` в первой строке
2. **YAML frontmatter:** `---\ndescription: ...\n---`

## Шаблон `SKILL.md`

### Legacy формат

```markdown
Description: <одна короткая фраза, ≤ 200 символов, с триггером срабатывания>

# Skill: <имя в kebab-case>

## Когда использовать

- <Триггер 1>
- <Триггер 2>

**Не использовать**, если ...

## Алгоритм

1. <Шаг 1>
2. <Шаг 2>
3. ...

## Пример

Вход: ...
Выход: ...

## Чего избегать

- <Антипаттерн 1>
- <Антипаттерн 2>
```

### YAML frontmatter формат

```yaml
---
name: <имя в kebab-case>
description: "<одна короткая фраза, ≤ 200 символов>"
risk: <low|medium|high|unknown>  # опционально
source: <community|official|internal>  # опционально
date_added: "YYYY-MM-DD"  # опционально
---

# Skill: <имя>

## When to Use
...
```

Поля `risk`, `source`, `date_added` — для документации, агент их не использует.

### Обязательные элементы

- **Описание** — `Description: ...` (legacy) или YAML `description: ...` (frontmatter). Идёт в системный промпт через `{{SKILLS_DESCRIPTION}}`.
- **Заголовок** `# Skill: <name>` — для человекочитаемости.
- **Раздел «Когда использовать»** — критерии срабатывания.
- **Раздел «Алгоритм»** — пошаговая инструкция.

### Желательные элементы

- Раздел «Когда **не** использовать» — отсечь похожие, но другие скиллы.
- Один–два примера (вход → выход).
- Раздел «Чего избегать» — типичные галлюцинации / ошибки.

## Правила именования

- **kebab-case**: `web-research`, `russian-style`, `code-review`.
- **Глагол + существительное** или **существительное-роль**: `summarize-text`, `email-draft`, `data-cleaning`.
- **Без аббревиатур**, если они не общеизвестны.
- **Описание ≤ 200 символов**, без воды, с триггером (по каким признакам агент должен этот скилл выбрать).

## Что НЕ кладём в `_skills/`

- **Системный промпт** — это `_prompts/agent_system.md`, отдельная сущность.
- **Описание tools** — оно автогенерируется из реестра, не дублируем.
- **Документацию проекта** — она в `_docs/`. Скилл — это **рабочая инструкция** агенту, не справка человеку.
- **Код** — никаких `.py` / `.js` / shell-скриптов. Только markdown.

## Жизненный цикл

1. Создаёшь `_skills/<name>/SKILL.md`.
2. Перезапускаешь процесс — `SkillRegistry` подхватит на старте.
3. Описание попадёт в системный промпт автоматически.
4. Агент сам решит, нужен ли этот скилл; если да — вызовет `load_skill("<name>")`.

> Hot-reload (правка без рестарта) — будущая фича, см. `_docs/roadmap.md` Этап 15.
