# MVP

> Этот документ описывает scope **первого функционального спринта** проекта (Спринт 01 «MVP Agent»). Спринт 00 (Bootstrap) — инфраструктурный, не имеет своего MVP-чек-листа: его критерии приёмки лежат в самом файле `_board/sprints/00-bootstrap.md`.

## 1. Цель MVP

Запускаемый Telegram-бот на Python/aiogram, который **не просто отвечает** на текст, а решает задачу пользователя в виде **агента**:

1. Получает задачу.
2. Думает (`thought`).
3. Выбирает инструмент (`action` + `args`) или сразу даёт финальный ответ.
4. Получает результат инструмента (`observation`).
5. Повторяет 2–4 до получения финального ответа или достижения лимита шагов.

LLM — локальная (Ollama, `qwen3.5:4b` для чата, `nomic-embed-text` для эмбеддингов, `llava:7b` для описания изображений). Долгосрочная память (саммари сессий) — в `sqlite-vec`.

Сценарий, на котором проверяем MVP, — три задачи из ТЗ:

- «Посчитай (123 + 456) * 2» → должен вызвать `calculator`, вернуть `1158`.
- «Прочитай файл и скажи, сколько там строк» → должен вызвать `read_file`, посчитать строки, вернуть число.
- «Сделай запрос и верни результат» → должен вызвать `http_request` (или `web_search` — в зависимости от формулировки), вернуть тело ответа / список найденных ссылок.

## 2. Что входит в MVP (Спринт 01)

### 2.1 Функциональность

- [ ] Запуск через polling (не webhook).
- [ ] Команда `/start` — приветствие + краткая инструкция.
- [ ] Команда `/help` — список команд и как пользоваться ботом.
- [ ] Команда `/models` — список доступных LLM (`OLLAMA_AVAILABLE_MODELS`).
- [ ] Команда `/model <name>` — выбор активной LLM для пользователя.
- [ ] Команда `/prompt <text>` — установка системного промпта на сессию (или сброс к default из `_prompts/agent_system.md`, если без аргумента).
- [ ] Команда `/new` — архивирует текущую сессию: суммирует историю, режет на чанки, сохраняет в `sqlite-vec` с эмбеддингом, очищает in-memory историю и стартует новую сессию (`conversation_id` ротируется).
- [ ] Команда `/reset` — очищает in-memory историю + per-user настройки. **Архив (`sqlite-vec`) не трогает**.
- [ ] Обработка произвольного текста как задачи для агента → запуск цикла → ответ пользователю.
- [ ] Индикатор «печатает…» (`ChatAction.TYPING`) во время работы цикла.
- [ ] Обработка ошибок LLM (недоступность, таймаут, 4xx/5xx) с понятным сообщением.
- [ ] Логирование в консоль и в файл (с ротацией).
- [ ] Логирование каждого шага цикла: `step=<n> kind=thought|action|final tool=<name> dur_ms=<n>`.

### 2.2 Tools (минимальный набор из ТЗ)

- [ ] `calculator` — безопасное вычисление арифметического выражения (без `eval`).
- [ ] `read_file` — чтение файла по путям из настраиваемого whitelist'а каталогов (по умолчанию — `data/`).
- [ ] `http_request` — GET-запрос к URL, возврат статуса и тела (с лимитом размера).
- [ ] `web_search` — DuckDuckGo через `ddgs`, top-N результатов в виде `[{title, href, snippet}]`.

### 2.3 Память

- [ ] Краткосрочная: in-memory история per-user (`HISTORY_MAX_MESSAGES`, FIFO + LLM-суммаризация при `HISTORY_SUMMARY_THRESHOLD`).
- [ ] Долгосрочная: `sqlite-vec` (`MEMORY_DB_PATH`). Поиск по саммари — отдельным инструментом `memory_search` (доступен агенту в цикле).

### 2.4 Skills (заготовка системы)

- [ ] Каталог `_skills/` со схемой `<name>/SKILL.md` (первая строка с `Description: ...`).
- [ ] Tool `load_skill(name)` — читает соответствующий `SKILL.md` и возвращает агенту его тело как observation.
- [ ] В системный промпт инжектируется список доступных скиллов с их описаниями (только первая строка), чтобы агент знал, что вообще существует.
- [ ] В MVP создан **минимум один пример-скилл** для smoke-проверки (например, `_skills/example-summary/SKILL.md`).

### 2.5 Качество

- [ ] Весь I/O — асинхронный (`async/await`).
- [ ] Нет необработанных исключений (глобальный error handler).
- [ ] Покрытие unit-тестами ключевой логики (агентный цикл, парсер JSON, реестр tools, sqlite-vec слой через моки и temp-БД).
- [ ] `.env.example` в репозитории, реальный `.env` — в `.gitignore`.
- [ ] `.gitignore` включает секреты, логи, виртуальное окружение, кэш, `data/`, `*.db`.
- [ ] `README.md` с инструкцией запуска, командами, описанием цикла и ограничений.

## 3. Что НЕ входит в MVP (явно отложено)

- **Planner** и **Critic** агенты (multi-agent pipeline) — отдельный спринт, см. `roadmap.md` Этап 4.
- **Web-адаптер** и **MAX-адаптер** — `roadmap.md` Этап 5.
- **Webhook**-режим Telegram — `roadmap.md` Этап 6.
- **Облачные LLM** (OpenAI, Anthropic и пр.) — запрещено по принципу `local-first` (`requirements.md` CON-2).
- **Файловые загрузки** в Telegram (фото, аудио, документы) — `roadmap.md` Этап 7.
- **Стриминг ответа** LLM в Telegram — `roadmap.md` Этап 8.
- **Throttling middleware** (rate-limit на пользователя) — `roadmap.md` Этап 9.
- **Docker / docker-compose** — `roadmap.md` Этап 10.
- **CI** (GitHub Actions) — `roadmap.md` Этап 11.

## 4. Модель по умолчанию

- LLM: `qwen3.5:4b` (из ТЗ).
- Embedding: `nomic-embed-text` (768 dimensions; настраивается через `EMBEDDING_MODEL` и `EMBEDDING_DIMENSIONS`).

> Точные имена тегов могут отличаться в вашей установке Ollama. При настройке проверяйте через `ollama list`. Имена моделей конфигурируются через `.env` (см. `stack.md` и `_docs/requirements.md`).

## 5. Критерии приёмки MVP (Спринт 01)

MVP считается готовым, если одновременно выполнены все пункты:

1. **Запуск**: `python -m app` стартует бота, polling работает, в логе появляется `Bot started`.
2. **/start**: команда отвечает приветственным сообщением со списком команд.
3. **Агент решает калькуляторную задачу**: «Посчитай (123 + 456) * 2» → в логе видны шаги цикла (минимум `thought`, `action: calculator`, `observation`, `final_answer`), пользователь получает `1158`.
4. **Агент решает файловую задачу**: «Прочитай файл `data/sample.txt` и скажи, сколько там строк» → в цикле виден `action: read_file`, ответ пользователю содержит число строк.
5. **Агент решает поисковую задачу**: «Найди, кто выиграл последний этап F1» → в цикле виден `action: web_search` (или `http_request`), ответ пользователю содержит осмысленное резюме.
6. **/new**: после нескольких сообщений отправить `/new` → в `data/memory.db` появляется хотя бы один чанк саммари (проверяется тестом или ручным запросом по БД).
7. **memory_search**: после `/new` агент в новой сессии может найти прежнюю информацию (запрос «что мы обсуждали раньше про X» → `action: memory_search`).
8. **Выбор модели**: `/models` показывает список, `/model <name>` переключает (если в `OLLAMA_AVAILABLE_MODELS` несколько).
9. **Ошибки**: при остановленной Ollama бот не падает, пользователю приходит сообщение об ошибке, в логе — запись уровня ERROR.
10. **Защита от бесконечного цикла**: если LLM не может прийти к `final_answer`, цикл останавливается на `AGENT_MAX_STEPS` шаге, пользователь получает сообщение «Агент не смог решить задачу за N шагов, попробуйте переформулировать».
11. **Логи**: файл логов создаётся, содержит записи о входящих сообщениях, шагах цикла, вызовах LLM и tools.
12. **Секреты**: в репозитории отсутствует `.env` с реальным токеном; есть `.env.example`.
13. **Тесты**: `pytest -q` проходит, покрыты как минимум:
    - Парсер JSON-ответа модели (валидные / битые / неожиданные форматы).
    - Агентный цикл (успех на 1 шаге, успех через несколько шагов, обрыв по лимиту).
    - Реестр tools (регистрация, валидация args, ошибки tool).
    - Каждый tool из MVP-набора.
    - `sqlite-vec` слой: запись чанка с эмбеддингом → KNN-поиск возвращает свой чанк.
    - Команды `/start`, `/new`, `/reset` (с моком LLM).
14. **README.md**: содержит разделы «Установка», «Настройка», «Запуск», «Команды бота», «Структура», «Тесты», «Ограничения».

## 6. Definition of Done / Definition of Ready

См. общие правила в `_board/plan.md` § «Правила работы со спринтами / задачами» и шаблон DoD в `_board/plan.md` § «Шаблон новой задачи».

В MVP-контексте:

- **DoR** (задача готова к работе): сценарий описан в `requirements.md`/`commands.md`/`agent-loop.md`/`tools.md`/`memory.md`; понятен затрагиваемый модуль (см. `project-structure.md`).
- **DoD** (задача закрыта): код реализован, проходит линтер/форматтер, добавлены/обновлены unit-тесты, `pytest -q` зелёный, документация обновлена при изменении поведения, ручная проверка в Telegram прошла.
