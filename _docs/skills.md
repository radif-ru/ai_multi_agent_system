# Skills

Документ описывает формат `_skills/`, как агент решает их применять и порядок добавления нового скилла. Связанные документы: `architecture.md` §3.8, `tools.md` §6, `prompts.md` §3.

## 1. Когда tool, а когда skill

| Tool | Skill |
|------|-------|
| Действие, имеющее **детерминированный код** (вычисление, HTTP-запрос, поиск). | **Знание о том, как решать класс задач** (последовательность шагов, формат ответа, эвристики). |
| Реализуется в Python (`app/tools/<name>.py`). | Реализуется markdown-инструкцией (`_skills/<name>/SKILL.md`). |
| Не требует LLM при выполнении. | Активируется внутри агентного цикла; LLM сама применяет инструкцию из тела скилла. |
| Контракт: `name`, `args_schema`, `async run`. | Контракт: `Description: ...` в первой строке + markdown-тело. |

Грубое правило: если бы человеку для решения класса задач нужна была **инструкция на бумаге**, это skill. Если нужна **программа** — это tool.

## 2. Структура каталога

```
_skills/
├── README.md                    # шаблон / правила (этот документ ссылается на него)
├── <skill-name>/
│   └── SKILL.md
├── <another-skill>/
│   └── SKILL.md
└── ...
```

Имя подпапки — **это имя скилла**, по которому агент его вызывает (`load_skill("<name>")`). Имена — kebab-case (`ascii-art`, `data-cleaning`, `russian-style`).

## 3. Формат `SKILL.md`

Поддерживаются **два формата** описания скилла:

1. **Legacy:** `Description: <текст>` в первой строке
2. **YAML frontmatter:** `---\ndescription: ...\n---`

Оба формата взаимозаменяемы с точки зрения агента — `SkillRegistry` парсит оба.

### 3.1 Legacy формат

**Первая строка** — обязательно `Description: <одна фраза>` (≤ 200 символов). Это описание идёт в системный промпт агента в плейсхолдер `{{SKILLS_DESCRIPTION}}`.

```markdown
Description: Кратко переформулирует длинный текст в саммари из 3–5 пунктов.

# Skill: example-summary

## Когда использовать
...
```

### 3.2 YAML frontmatter формат

Файл начинается с `---`, затем YAML с полем `description`, затем закрывающий `---`. Остальное — тело скилла.

```yaml
---
name: bash-linux
description: "Bash/Linux terminal patterns. Critical commands, piping, error handling."
risk: unknown
source: community
date_added: "2026-02-27"
---

# Bash Linux Patterns

## When to Use
...
```

Дополнительные поля в frontmatter (`risk`, `source`, `date_added`) — опциональны, сохраняются для документации, но не используются агентом.

### 3.3 Структура тела скилла

После заголовка/разделителя идёт markdown-тело:

```markdown
# Skill: example-summary

## Когда использовать

- Пользователь прислал длинный текст и просит «сократить», «подытожить», «выделить главное».
- Длина исходника > 1000 символов.

**Не использовать**, если пользователь просит **переписать** текст или **перевести** на другой язык — это другие скиллы.

## Алгоритм

1. Прочитай текст целиком.
2. Выдели 3–5 ключевых тезисов.
3. Сформулируй каждый тезис одним предложением, без вводных слов.
4. Верни их в `final_answer` как маркированный список.

## Пример

Вход: `<длинный_текст>` …
Выход:
```
- Тезис 1.
- Тезис 2.
- Тезис 3.
```

## Чего избегать

- Не добавляй извинений / обращений / преамбул.
- Не вкладывай тезисы один в другой — только плоский список.
```

### 3.4 Обязательные элементы

- Описание скилла (в `Description:` или YAML `description`).
- Заголовок `# Skill: <name>` или аналог для человекочитаемости.
- Раздел «Когда использовать» — критерии срабатывания.
- Раздел «Алгоритм» — пошаговая инструкция.

### 3.5 Желательные элементы

- Раздел «Когда **не** использовать» — отсечь похожие, но другие скиллы.
- Один–два примера (вход → выход).
- Раздел «Чего избегать» — типичные галлюцинации / ошибки.

## 4. Как агент решает применить skill

1. На старте процесса `SkillRegistry` сканирует `_skills/`, парсит первую строку каждого `SKILL.md`.
2. Список вида `- <name>: <description>` подставляется в системный промпт агента (`{{SKILLS_DESCRIPTION}}`).
3. Когда LLM в `thought` решает «надо использовать skill X», она вызывает tool `load_skill` с `args={"name": "X"}`.
4. `LoadSkillTool` возвращает `observation` = тело `SKILL.md` без первой строки.
5. На следующем шаге LLM применяет инструкцию из тела и продолжает цикл.

Системный промпт **подсказывает** агенту использовать `load_skill` (см. `_prompts/agent_system.md`), но **не заставляет**: если задача простая и решается одним tool'ом, скилл подгружать не нужно.

## 5. Как добавить новый skill

1. Создать каталог `_skills/<name>/`.
2. Положить туда `SKILL.md` по формату §3.
3. Перезапустить процесс — `SkillRegistry` подхватит на старте.
4. Smoke-проверка: ручной диалог в Telegram, описать задачу, для которой нужен новый скилл; проверить, что в логах есть `tool=load_skill name=<name>` на каком-то из шагов цикла.

Тестов **на сам скилл** не пишем (это контент, а не код). Тесты на `SkillRegistry` (что он корректно парсит первую строку, перечисляет скиллы, отдаёт тело без первой строки) — единые.

## 6. Hot-reload (НЕ MVP)

Сейчас скиллы читаются один раз при старте процесса. Если поправил `SKILL.md` — нужен рестарт. Hot-reload (watcher на `_skills/`) — кандидат на отдельный спринт (`roadmap.md`).

## 7. Что НЕ кладём в `_skills/`

- **Системный промпт** — это `_prompts/agent_system.md`, отдельная сущность.
- **Описание tools** — оно автогенерируется из реестра, не дублируем в скиллах.
- **Документацию проекта** — она в `_docs/`. Скилл — это **рабочая инструкция** агенту, не справка человеку.
- **Код** — никаких `.py` / `.js` / shell-скриптов. Только markdown с инструкцией.

## 8. Примеры существующих скиллов

- **`example-summary`** — переформулирует длинный текст в саммари из 3–5 пунктов.
- **`bash-linux`** — bash/linux terminal patterns, критические команды, piping, error handling.
- **`bash-pro`** — продвинутый bash: скрипты, отладка, оптимизация.
- **`weather`** — получение погоды через wttr.in с fallback на веб-поиск.
- **`ocr`** — правила выбора между OCR (`ocr_image`) и Vision (`describe_image`) для обработки изображений.

## 9. Принцип «короткие описания»

Описание скилла (первая строка) видит **каждый** агентный цикл, даже если скилл не нужен. Поэтому:

- Описание ≤ 200 символов.
- Без воды («Этот скилл предназначен для…» → плохо; «Кратко переформулирует длинный текст в саммари из 3–5 пунктов» → хорошо).
- В описании сразу видно **триггер** срабатывания (по каким признакам агент должен этот скилл выбрать).

При большом числе скиллов системный промпт может разрастись и помешать слабым моделям. Тогда — переходим на **двухуровневую загрузку**: первая строка → второй файл `INDEX.md` (категории) → конкретные `SKILL.md`. Это roadmap-кандидат; в MVP не нужен.
