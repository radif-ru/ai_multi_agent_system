# Требования

Документ формализует требования к проекту в виде проверяемых FR / NFR / CON / ASM. Источник — постановка задачи (ТЗ пользователя), `mvp.md` (scope первого MVP), `architecture.md` (дизайн), `agent-loop.md`, `memory.md`, `tools.md`, `skills.md`.

## 1. Функциональные требования (FR)

### 1.1 Telegram и команды

| ID    | Требование |
|-------|------------|
| FR-1  | Бот принимает текстовые сообщения от пользователя в Telegram. |
| FR-2  | Получение апдейтов — через polling, **не webhook**. |
| FR-3  | Поддерживается команда `/start` с приветственным сообщением. |
| FR-4  | Поддерживается команда `/help` с подробной справкой. |
| FR-5  | Поддерживается команда `/models` — список доступных LLM. |
| FR-6  | Поддерживается команда `/model <name>` — переключение активной LLM. |
| FR-7  | Поддерживается команда `/prompt [<text>]` — установка / сброс системного промпта на сессию. |
| FR-8  | Поддерживается команда `/new` — архивирование текущей сессии и старт новой (см. FR-21..FR-23). |
| FR-9  | Поддерживается команда `/reset` — очистка in-memory истории и сброс per-user настроек. **Не трогает архив.** |
| FR-10 | Произвольный текст (не команда) запускает агентный цикл; результат возвращается пользователю. |

### 1.2 Агентный цикл

| ID    | Требование |
|-------|------------|
| FR-11 | Цикл работает по схеме `thought → action → observation → thought → ...` до получения `final_answer` или достижения лимита шагов. См. `agent-loop.md`. |
| FR-12 | LLM возвращает структурированный ответ строго в JSON-формате одного из двух видов: `{"thought": ..., "action": ..., "args": {...}}` либо `{"final_answer": ...}`. |
| FR-13 | Цикл ограничен количеством шагов (`AGENT_MAX_STEPS`, default 10); при превышении пользователь получает сообщение «Агент не смог решить задачу за N шагов». |
| FR-14 | На каждый шаг цикла пишется лог-строка `INFO step=<n> kind=thought\|action\|final user=<id> tool=<name> dur_ms=<n>`. |
| FR-15 | Любой невалидный JSON ответа модели — обрабатывается как `LLMBadResponse` и не валит цикл; пользователю отдаётся понятное сообщение. |

### 1.3 Tools

| ID    | Требование |
|-------|------------|
| FR-16 | Реализован tool `calculator` — безопасное вычисление арифметического выражения (без `eval`/`exec`). |
| FR-17 | Реализован tool `read_file` — чтение файла по путям из whitelist каталогов (по умолчанию `data/`); пути вне whitelist отклоняются. |
| FR-18 | Реализован tool `http_request` — HTTP GET по URL, ограничение размера ответа, таймаут. |
| FR-19 | Реализован tool `web_search` — поиск через DuckDuckGo (`ddgs`), top-N результатов в виде `[{title, href, snippet}]`. |
| FR-20 | Реализован tool `memory_search` — KNN-поиск по семантической памяти (`sqlite-vec`), возвращает top-K чанков с дистанцией. |

### 1.4 Память

| ID    | Требование |
|-------|------------|
| FR-21 | Краткосрочная память: каждый пользователь имеет независимую in-memory историю диалога. История ограничена по размеру (`HISTORY_MAX_MESSAGES`, default 20) и при превышении порога (`HISTORY_SUMMARY_THRESHOLD`, default 10) суммируется. Персистентного хранения **сырого диалога** нет — теряется при рестарте процесса. |
| FR-22 | Долгосрочная память: при `/new` старая сессия суммируется LLM, режется на чанки (`MEMORY_CHUNK_SIZE` символов с перекрытием `MEMORY_CHUNK_OVERLAP`), для каждого чанка получается embedding (`EMBEDDING_MODEL`), чанк + вектор + метаданные пишутся в `sqlite-vec` (`MEMORY_DB_PATH`). |
| FR-23 | Архив **не зачищается** командой `/reset`; команда `/new` его пополняет, но не очищает. Очистка архива — внешняя процедура (например, удаление `.db`-файла). |
| FR-24 | Tool `memory_search` доступен агенту в любой сессии; он ищет в архиве по семантическому сходству и возвращает top-K чанков. |

### 1.5 Skills и Prompts

| ID    | Требование |
|-------|------------|
| FR-25 | Каталог `_skills/<name>/SKILL.md`. Первая строка `SKILL.md` — `Description: <одна фраза>`, остальное — markdown-инструкция. См. `skills.md`. |
| FR-26 | В системный промпт агента инжектируется список доступных скиллов в формате `- <name>: <description>` (только первая строка каждого `SKILL.md`). |
| FR-27 | Tool `load_skill(name)` читает соответствующий `SKILL.md` и возвращает агенту его содержимое (без первой строки) как `observation`. |
| FR-28 | Системный промпт агента и промпт суммаризации хранятся в `_prompts/` как markdown-файлы; путь главного промпта задаётся через `AGENT_SYSTEM_PROMPT_PATH` в `.env`. См. `prompts.md`. |

### 1.6 Конфигурация и логирование

| ID    | Требование |
|-------|------------|
| FR-29 | Все секреты и настройки загружаются из `.env` через `pydantic-settings`. |
| FR-30 | В репозитории есть `.gitignore`, исключающий `.env`, логи, кэши, venv, `data/`, `*.db`. |
| FR-31 | В репозитории есть `README.md` с инструкцией запуска и описанием команд. |
| FR-32 | Все запросы к LLM логируются (`user`, `chat`, `model`, `step`, `len_in`, `len_out`, `dur_ms`, `status`). |
| FR-33 | Перед каждым LLM-запросом полный контекст (`messages`) и его размер (в приближённых токенах) логируются. Печать полного payload управляется флагом `LOG_LLM_CONTEXT`. |

### 1.7 Медиа-файлы

| ID    | Требование |
|-------|------------|
| FR-34 | Бот поддерживает загрузку файлов из Telegram с проверкой размера (`TELEGRAM_MAX_FILE_MB`, default 20). При превышении лимита — понятное сообщение пользователю. |
| FR-35 | Handler `Document` обрабатывает документы (PDF/TXT/MD). Скачивает файл, формирует обогащённый goal с путём к файлу и caption, передаёт в агентный цикл. Агент использует tool `read_document`. |
| FR-36 | Tool `read_document` читает документы из временной директории (`TMP_FILES_DIR`, default `tmp`). Поддерживает PDF (через `pypdf`), TXT, MD. Защита от path traversal. Усечение вывода до `max_chars` (default 8000). |
| FR-37 | Handler `Voice` обрабатывает голосовые сообщения (Voice/Audio). Скачивает файл, транскрибирует через `Transcriber` (faster-whisper), передаёт распознанный текст в агентный цикл. Опциональная зависимость — при недоступности fallback-сообщение. |
| FR-38 | Handler `Photo` обрабатывает фотографии. Скачивает файл, описывает через `Vision` (Ollama vision API), передаёт описание в агентный цикл. Опциональная зависимость — при недоступности vision-модели fallback-сообщение. |
| FR-39 | Временные файлы удаляются после обработки. Директория `TMP_FILES_DIR` создаётся автоматически при старте. |

## 2. Нефункциональные требования (NFR)

| ID     | Требование |
|--------|------------|
| NFR-1  | Стек: **Python 3.11+** + **aiogram 3** + **ollama** + **sqlite-vec**. |
| NFR-2  | Весь код I/O — асинхронный (`async/await`). |
| NFR-3  | Нет необработанных исключений: все ошибки логируются и пользователю возвращается понятное сообщение. |
| NFR-4  | Код покрыт автоматическими тестами (`pytest`); см. `testing.md` § «Покрытие». |
| NFR-5  | Из персистентного хранения используется **только** `sqlite-vec` БД для семантической памяти (`MEMORY_DB_PATH`). Другие БД, файлы или Redis не вводим. |
| NFR-6  | Нет использования облачных LLM. |
| NFR-7  | Бот стабильно обрабатывает несколько последовательных / параллельных запросов. |
| NFR-8  | Логи пишутся в консоль и в файл (с ротацией); файл логов — в `.gitignore`. |
| NFR-9  | Архитектура — слоистая: handler не знает про HTTP, агентный цикл не знает про aiogram, tool не знает про LLM. См. `architecture.md` §3. |
| NFR-10 | Архитектура **расширяема** под мульти-агент: добавление Planner / Critic не должно требовать переписывания Executor / tools / memory; см. `architecture.md` §7. |
| NFR-11 | Архитектура **расширяема** под новые адаптеры (web, MAX, и др.): доменная модель не зависит от Telegram, протокол `task.create` (см. `architecture.md` §3.10) — единая точка входа. |

## 3. Ограничения (CON)

| ID    | Ограничение |
|-------|-------------|
| CON-1 | Запрещено персистентное хранение **сырого** диалога (полный текст user/assistant сообщений в БД). В долгосрочную память пишется **только саммари** в виде чанков. См. `memory.md` §4. |
| CON-2 | Запрещены облачные LLM (OpenAI, Anthropic, Google, Mistral API и т. п.). |
| CON-3 | Запрещены другие БД, кроме `sqlite-vec` (один `.db`-файл). Никаких PostgreSQL, MongoDB, Redis. |
| CON-4 | Запрещён webhook (только polling) в MVP. Webhook — кандидат на отдельный спринт (см. `roadmap.md` Этап 6). |
| CON-5 | Запрещена работа агента в обход системного промпта: ответ модели, не соответствующий JSON-схеме из `agent-loop.md` §2, считается ошибкой и обрабатывается единообразно. |

## 4. Предположения (ASM)

| ID    | Предположение |
|-------|---------------|
| ASM-1 | У разработчика локально установлен и запущен Ollama на стандартном порту. |
| ASM-2 | Модели, указанные в `.env` (`OLLAMA_AVAILABLE_MODELS` и `EMBEDDING_MODEL`), заранее скачаны (`ollama pull <model>`). |
| ASM-3 | Telegram-бот создан через `@BotFather` и токен доступен. |
| ASM-4 | Per-user runtime-состояние (текущая модель, системный промпт, in-memory история) живёт только в памяти процесса. После рестарта возвращается к default из `Settings`. |
| ASM-5 | Используется Python 3.11+. |
| ASM-6 | Архив семантической памяти переживает рестарт процесса (это не in-memory). При удалении `.db`-файла архив теряется. |

## 6. Критерии верификации

Каждое требование должно быть подтверждено как минимум одним из способов:

- **T** — автоматический тест (`pytest`).
- **M** — ручная проверка по сценарию (например, в Telegram).
- **I** — инспекция кода / репозитория (например, наличие `.env.example`, `.gitignore`).

Сводная таблица составляется по факту реализации в Спринте 01 и обновляется в `_board/sprints/01-mvp-agent.md` § DoD каждой задачи.
