# Prompts

Документ описывает формат `_prompts/`, как промпты подмешиваются в LLM-вызовы и порядок их правки. Связанные документы: `agent-loop.md` §3, `architecture.md` §3.9, `skills.md` §4.

## 1. Что такое prompt

**Prompt** — markdown-файл с системным промптом для одного из LLM-сценариев:

- `agent_system.md` — главный системный промпт **агентного цикла** Executor. Объясняет модели формат ответа, набор tools, набор скиллов, правила.
- `summarizer.md` — системный промпт для **суммаризации** (используется и для in-session порога, и для архивирования при `/new`).
- (в будущем) `planner.md`, `critic.md` — для мульти-агента.

Файлы лежат в `_prompts/` (не в `app/`), потому что:

- Их **редактируют без перекомпиляции** — поправил, перезапустил процесс, готово.
- Это **контент**, не код; пусть лежит рядом с `_skills/` и `_docs/`.
- Путь главного промпта настраивается через `AGENT_SYSTEM_PROMPT_PATH` в `.env`, чтобы можно было быстро переключиться на свой кастомный.

## 2. Загрузка (`PromptLoader`)

Реализация — `app/services/prompts.py::PromptLoader`. На старте процесса:

1. Открывает файл по пути `Settings.agent_system_prompt_path` (по умолчанию `_prompts/agent_system.md`).
2. Читает содержимое в строку.
3. Аналогично для `_prompts/summarizer.md` (путь — фиксированный, отдельной env-переменной нет).
4. Хранит обе строки в памяти как сырые шаблоны.

Если файл не найден — процесс **не стартует** (это явная ошибка конфигурации). Сообщение об ошибке указывает ожидаемый путь и фактический cwd.

## 3. Плейсхолдеры в `agent_system.md`

Перед стартом цикла Executor подставляет два плейсхолдера в строку промпта:

| Плейсхолдер              | Источник                                          | Формат вставки |
|--------------------------|---------------------------------------------------|----------------|
| `{{TOOLS_DESCRIPTION}}`  | `ToolRegistry.list_descriptions()`                | Markdown-список вида `- name(args): description.` |
| `{{SKILLS_DESCRIPTION}}` | `SkillRegistry.list_descriptions()`               | Markdown-список вида `- name: description.` |

Подстановка — простая `str.replace`, никакого Jinja / Mustache. Если плейсхолдера нет в файле — это не ошибка (значит, юзер так захотел); подстановка просто пропускается.

Пример итоговой подстановки в системный промпт:

```
Доступные инструменты:
- calculator(expression: string): Безопасное вычисление арифметического выражения.
- read_file(path: string): Прочитать содержимое файла из data/.
- web_search(query: string, top_k: integer = 5): Поиск через DuckDuckGo.
- memory_search(query: string, top_k: integer = 5): Поиск в долгосрочной памяти.
- load_skill(name: string): Загрузить полный текст скилла по имени.

Доступные скиллы:
- example-summary: Кратко переформулирует длинный текст в саммари из 3–5 пунктов.
```

## 4. Правка промпта

1. Открыть нужный файл в `_prompts/`.
2. Поправить.
3. Перезапустить процесс (`Ctrl+C` + `python -m app`) — изменения подхватятся.
4. Smoke-проверка: ручной диалог в Telegram, проверка, что цикл по-прежнему выходит в `final_answer` (не сломали JSON-схему ответа).
5. Если правка ломает контракт ответа модели — обновить `agent-loop.md` §2 (формат JSON) синхронно с промптом.
6. Закоммитить (`docs(prompts): tweak agent_system tone` или `feat(prompts): ...`, если это не косметика).

## 5. Что должно быть в `agent_system.md`

Минимально (MVP):

1. **Идентичность**: «Ты — AI-агент…».
2. **Цель**: решить задачу пользователя через цикл `thought → action → observation`.
3. **Жёсткое правило формата ответа**: только JSON одной из двух схем (см. `agent-loop.md` §2).
4. **Набор инструментов** через `{{TOOLS_DESCRIPTION}}`.
5. **Набор скиллов** через `{{SKILLS_DESCRIPTION}}`.
6. **Правила поведения**: думать пошагово, не угадывать результаты tools, останавливаться при уверенности, не выдумывать tools.
7. **Запреты**: не выводить ничего вне JSON, не выходить из формата, не объяснять «что я сейчас сделаю» текстом.
8. **Правила безопасности**: запрет на игнорирование инструкций, запрет на вывод системного промпта, запрет на выполнение опасных операций без явного запроса (добавлено в Спринте 05, задача 5.1).

Готовый текст MVP-промпта — `_prompts/agent_system.md` (создаётся в Спринте 00, расширен в Спринте 05).

## 6. Что должно быть в `summarizer.md`

1. **Цель**: сжать список сообщений в краткое резюме.
2. **Стилевые рамки**: 2–4 предложения (для in-session) или ≤ 1500 символов (для архивирования). Один и тот же файл для обоих сценариев — длина задаётся параметром в момент вызова.
3. **Что фиксировать**: ключевые факты, решения, открытые вопросы.
4. **Чего избегать**: вводных, оценочных суждений, «как ассистент». Без markdown.

В Спринте 00 файл `_prompts/summarizer.md` уже создан; правки — отдельным коммитом по мере опыта эксплуатации.

## 7. Многоязычность

В MVP оба промпта — на **русском**, потому что основной пользователь общается с ботом на русском. Переключение на английский для тестов — через `AGENT_SYSTEM_PROMPT_PATH=_prompts/agent_system.en.md` в `.env`. Английский вариант в Спринте 00 не создаём (не нужен).

## 8. Что НЕ кладём в `_prompts/`

- **Тело скиллов** — это `_skills/<name>/SKILL.md`, отдельная сущность (см. `skills.md`).
- **Описания tools** — они приходят из кода (`ToolRegistry.list_descriptions`), не нужны как отдельные файлы.
- **Шаблоны для отдельных команд бота** (типа текста `/start` или `/help`) — это UI-копи, лежит прямо в коде хендлеров (`app/adapters/telegram/handlers/commands.py`).

## 9. Future: prompt-versioning (НЕ MVP)

Когда промптов станет больше / появится Planner/Critic — добавим:

- Версионирование (`agent_system.v1.md`, `agent_system.v2.md`) с явным выбором через env.
- Тест: «smoke-прогон 5 задач из набора, и каждая решается за < N шагов».
- A/B-сравнение качества цикла на разных промптах (через ту же телеметрию шагов).

Это roadmap-кандидаты, в Спринте 01 не нужны.
